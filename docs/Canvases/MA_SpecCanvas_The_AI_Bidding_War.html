1 System Intent
1
🎯 System Mission
Simulate a multi-party procurement negotiation in which a buyer and three competing vendors pursue conflicting economic goals under private information (budgets, floors, scenario-specific stressors). The collective objective is not a single model’s summary deal — it is to produce traceable negotiation outcomes (award, NO_AWARD, or guardrail block) while demonstrating that governance (isolation, audit-before-approval, deterministic spend enforcement) can coexist with LLM-generated dialogue. End users in the narrative are CPOs and procurement leaders evaluating agentic stacks for policy-bound sourcing; the lab artifact is a reproducible CLI simulation with logged evidence.
2
👥 Agent Roster & Profiles
Agents Buyer — evaluates bids and may emit AWARD_JSON or continue; optional malicious buyer persona per scenario. Vendors A, B, C — distinct premium / budget / mid-tier roles with asymmetric floors. Strategic Advisor — in-run governance; not a bidder; audits proposed awards vs RFP and full stitched threads. Human operator — gatekeeper at terminal for execution approval. CFO narrative (offline) — optional batch report over evidence logs, not a live negotiator.
Profiling Designer-authored agent.md personas + orchestrator-built prompts; single Claude model family via ANTHROPIC_MODEL; Advisor uses ADVISOR_SYSTEM_PROMPT in agents/advisor.py.
Justify Separate agents preserve adversarial structure and information asymmetry; merging into one thread would collapse negotiation into self-dialogue. Advisor is separate from buyer so assurance is not confounded with bidding. Human gate is required for authorized spend in the simulation narrative.
3
🔲 System Scope & Boundaries
In scope Round-based dialogue (up to three rounds); RFP input (CLI or scenario text); 11 scripted scenarios in config/scenarios.json; structured logs and evidence JSON; Strategic Audit + human approval before award_contract; optional governance nudge after veto; evaluation harvest to CSV and FINAL_EVALUATION_REPORT.md.
Why MA Procurement is inherently multi-principal with private constraints; isolation of vendor threads models competitive bidding without cross-leakage — the problem cannot be faithfully represented as one omniscient assistant.
Out scope Live supplier APIs, real payments, legal contract generation, enterprise IAM, formal latency SLOs, unbounded negotiation length, cross-session persistent memory for negotiators, production integration.
2 Multi-Agent Structure
4
🏗️ Social & Organizational Structure
Hierarchy Orchestrator (orchestrator.py) schedules all LLM calls and parses structured lines; it does not “negotiate” in prose — it enforces protocol. Buyer is the hub decision-maker for awards. Vendors are peers in competition, not a team. Advisor reports to the human on governance, not to vendors.
Teams Vendors A→B→C run in a fixed sequence each round (not a cooperative sub-team). Buyer operates after all bids for that round are parsed. No peer-to-peer vendor messaging.
Authority Human can veto execution of a proposed award. award_contract (Python) rejects any price above the skill’s $15,000 hard limit regardless of LLM prose — non-negotiable. Vendors cannot edit buyer or each other’s outputs.
Trust Vendors are not trusted with competitors’ raw threads. Buyer consolidated view uses parsed BID_JSON only. The same buyer assistant text is relayed to every vendor — trust that it does not restate other vendors’ commercial terms is enforced by prompt + runtime append (standard) or relaxed for red-team (malicious default). Strategic Audit + human gate form the trust boundary before spend. Advisor failure produces a logged stub.
5
📡 Communication Structure & Protocol
Topology Hub-and-spoke: orchestrator mediates every vendor and buyer API call; no vendor-to-vendor channel.
Messages Natural language plus machine-parseable lines: BID_JSON:, AWARD_JSON:, optional CONTINUE / NO_AWARD conventions; evidence JSON for tooling.
Pattern Turn-based rounds within a single Python process; synchronous API calls; terminal prompts for human gate.
6
🌐 Environment & Interface Structure
Type Sandbox simulation: CLI session, Anthropic API as the only external model dependency; optional Docker for packaging.
Shared In-session orchestrator state (vendor_histories, conversation_buyer, buyer_reply_prev); append-only logs/ during the run; no shared vector DB across agents.
Boundary Cold-start inputs: .env, scenarios.json, personas — not prior logs as control inputs. Prior session logs are for offline evaluation and CFO-style reporting only.

3 Multi-Agent Behavior & Control
7
🧩 Individual Behavior Specification
Spatial Vendors: Anthropic text only; no tools. Buyer: same; may propose awards. Advisor: reads stitched threads for audit only. Execution: skills/award_contract.py only after human yes.
Temporal Per round: vendors A then B then C, then buyer; max three rounds; optional finalization segment if no contract; post-veto nudge may loop buyer without new vendor calls until rules advance the session.
Cognitive Each turn is a single LLM completion under orchestrator control (bounded, not unbounded ReAct). Policy enforcement for spend is not delegated to model reasoning — deterministic code.
8
🔄 Collective & Social Behavior
Planned Competitive bidding with buyer counter-pressure; audit and human gate on award path; optional nudge to steer buyer after veto.
Emergent LLM phrasing and concession patterns vary run-to-run; outcomes must still map to rubric baselines and harvested fields.
Social Dominant stance: competitive (vendors vs buyer and each other for selection); cooperation only in the weak sense of following protocol.
9
📜 Operating Rules, Norms & Human Checkpoints
Norms Evidence-first evaluation: claims about runs must be traceable to logs/CSVs. Isolation is a norm: no vendor sees another’s raw thread.
Rules Executable spend: skills/award_contract.py enforces a fixed BUYER_HARD_LIMIT of $15,000 (not parameterized per call). The orchestrator still loads per-scenario hard_budget_ceiling from config/scenarios.json for prompts and operator messaging — narrative may reference other figures, but over-$15k awards are rejected by the skill. Max rounds; parseable JSON lines for harvest; Advisor API error → logged stub.
Termination Stop on successful guarded contract, explicit walk-away, round exhaustion, or finalization path completing with award or NO_AWARD.
Gated Human yes/no after Strategic Audit before any award_contract execution; re-audit when buyer submits a new AWARD_JSON after nudge.
Denied Over-ceiling execution; structural cross-vendor raw-bid leakage (threads are isolated). Leakage via shared buyer prose is reduced by policy, not eliminated without Phase 3 per-vendor text. Unilateral LLM “contract” without guardrail; external email/API calls from vendor agents in v1.
10
♻️ Cross-Domain Feedback Topology
Reinforcing Competitive pressure can drive bids and buyer demands in successive rounds — amplifying movement toward a deal or a hard stop (e.g. NO_AWARD) unless capped by rounds.
Balancing Hard ceiling guardrail, human veto, Strategic Audit critique, and round limit damp runaway “verbal” awards or unsafe spend.
Dominance Under red-team or over-budget scenarios, balancing loops dominate (block / veto). Under cooperative negotiation scenarios, reinforcing price movement may dominate until governance checks fire.
Cross-domain Governance domain (audit Markdown, human choice) intersects economic domain (price, vendor id): audit informs the human; only award_contract commits spend — separating rhetoric from execution.
4 Context & Evaluation
11
💾 System I/O & Shared State
IN Scenario id and config; optional pasted RFP; operator responses at human gate and optional nudge text; .env API configuration.
OUT Terminal transcript; scenario_*.log; evidence_log_*.json; optional contract.txt; exported evaluation tables and FINAL_EVALUATION_REPORT.md; optional CFO Markdown in CFO Reports/.
Shared Orchestrator-maintained hub state and evidence accumulator for the session; evaluation artifacts derived post-hoc from logs.
Private Per-vendor thread histories visible only to orchestrator + that vendor’s prompts; buyer thread is buyer-only until stitched for Advisor.
12
⚠️ System Failure Modes & Escalation
Cascade Persuasive buyer or vendor prose could encourage a bad in-budget award — mitigated by Strategic Audit + human gate; over-ceiling cannot cascade to execution.
Deadlock Stalemate scenarios (e.g. scenario 5 design intent); bounded rounds prevent infinite loops; session may end NO_AWARD.
Herd Reduced by thread isolation; buyer still sees aggregated bids — evaluation checks negotiation quality via rubric. Shared buyer_reply_prev means vendors see identical buyer language; public-broadcast rules address cross-price inference for standard mode.
Escalate Human gate for spend; Advisor error stub surfaces failure without hiding the audit slot; documented NO_AWARD paths in logs for stress scenarios.
13
🎲 Scenario & Uncertainty Management
Uncertainty Model stochasticity in dialogue; parameter uncertainty bounded by scenario JSON (budgets, ceilings, modes). Not probabilistic calibration of real markets.
Scenarios 11 scripted IDs covering baseline, guardrail stress, finance squeeze, quality/compliance, stalemate, compromised gatekeeper, and boundary cases — selected by course design and Phase 1 feedback; operator chooses --id.
Communication Audit surfaces risks and gaps in Markdown; CSV harvest exposes Status, reasons, rounds, savings — uncertainty in “best deal” is acceptable where rubric allows NO_AWARD with documented trace.
Purpose Explain and govern simulated outcomes: demonstrate traceable policy behavior (governance pass) while scoring negotiation success per scenario — not production forecasting.
14
✅ System Success Criteria
Individual Roles produce parseable bids/awards where required; Advisor produces audit sections or explicit error stub; guardrail rejects illegal prices deterministically.
Collective Multi-agent design justified vs single-thread chat: isolation + hub aggregation + governance layer — measured via rubric governance vs negotiation columns and harvest metrics in FINAL_EVALUATION_REPORT.md §3.2.
Emergent Variance across LLM runs is expected; outcomes remain classifiable against §2 baselines; failure/boundary cases documented in §3.3 and Appendix A.
Safety No over-ceiling SUCCESS; isolation and audit/HITL trails present in logs for all scenario IDs in the evaluation snapshot; human checkpoints before spend execution.
